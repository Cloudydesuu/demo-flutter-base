# Getting Started

## Supabase

**Supabase** là một nền tảng Backend-as-a-Service (BaaS) mã nguồn mở, ra mắt năm 2020 nhằm cung cấp giải pháp thay thế cho Firebase. Supabase được xây dựng trên nền tảng cơ sở dữ liệu PostgreSQL – nghĩa là sử dụng mô hình cơ sở dữ liệu quan hệ với hỗ trợ SQL đầy đủ, cho phép ACID và các truy vấn phức tạp. Khác với Firebase (NoSQL), Supabase mang đến trải nghiệm Firebase-like nhưng dựa trên các công nghệ chuẩn và mở (Postgres, RESTful API…), có thể tự self-host nếu muốn kiểm soát hạ tầng của riêng mình

## Tính năng chính

Kiến trúc và các tính năng chính: Supabase bao gồm một bộ các dịch vụ backend tương tự Firebase, bao phủ từ cơ sở dữ liệu, realtime đến xác thực người dùng và lưu trữ file

### Cơ sở dữ liệu (Database)

**Cơ sở dữ liệu (Database)**: Supabase sử dụng PostgreSQL làm database lõi. Điều này đem lại tính nhất quán mạnh và ACID, cho phép thiết kế các bảng quan hệ với schema rõ ràng. Nhà phát triển có thể tận dụng SQL đầy đủ (SELECT, JOIN, subquery, transaction nhiều bước) để truy vấn và xử lý dữ liệu phức tạp – điều vượt trội so với mô hình NoSQL của Firebase (vốn không hỗ trợ join và hạn chế truy vấn). Vì dựa trên Postgres, Supabase cũng hỗ trợ nhiều kiểu dữ liệu và indexing nâng cao (ví dụ: JSON, text search, GIS qua PostGIS, v.v.) để tối ưu hiệu năng cho ứng dụng

### Realtime

**Realtime**: Supabase cung cấp tính năng realtime giúp ứng dụng có thể nhận được cập nhật dữ liệu tức thì tương tự Firebase. Cụ thể, Supabase sử dụng PostgreSQL logical replication kết hợp với Phoenix Channels (websocket) để lắng nghe các thay đổi trong database. Ứng dụng có thể subscribe để nhận sự kiện khi bảng hoặc hàng dữ liệu có INSERT, UPDATE, DELETE ￼. Đáng chú ý, Supabase còn hỗ trợ sẵn cơ chế Presence (hiện diện) để theo dõi trạng thái người dùng trực tuyến, hỗ trợ tốt cho các ứng dụng realtime có tính tương tác cao (như collaborative editors, chat) Tuy realtime của Supabase mạnh mẽ và dựa trên dữ liệu nguồn trực tiếp từ DB, tính năng offline hiện chỉ ở mức cơ bản – client Supabase chưa tự đồng bộ offline mà cần lập trình viên tự xây dựng chiến lược caching nếu muốn hỗ trợ ngoại tuyến

### Xác thực người dùng (Auth)

**Xác thực người dùng (Auth)**: Supabase tích hợp dịch vụ Supabase Auth (dựa trên dự án mã nguồn mở GoTrue) để quản lý người dùng và xác thực. Supabase Auth hỗ trợ các phương thức đăng nhập phổ biến: email/mật khẩu, đăng nhập OAuth qua mạng xã hội (Google, GitHub, Facebook, Apple, v.v.), OTP qua SMS, cũng như đăng nhập không mật khẩu (magic link). Tính năng đa yếu tố (MFA) cũng đã được hỗ trợ tương tự Firebase. Điểm khác biệt là Supabase gắn liền hệ thống auth với Postgres – mọi tài khoản user đều lưu trong bảng PostgreSQL. Nhờ đó, Supabase tận dụng được Row Level Security (RLS) của Postgres để kiểm soát truy cập ở mức hàng dữ liệu: bạn có thể viết các chính sách bảo mật bằng SQL để quy định người dùng nào được phép SELECT/INSERT/UPDATE trên từng bảng hoặc thậm chí từng dòng Cách tiếp cận này cho phép phân quyền rất linh hoạt và chi tiết (tương đương tầng database), phù hợp với các ứng dụng yêu cầu mô hình quyền phức tạp. (Trong khi đó Firebase sử dụng security rules riêng cho từng dịch vụ – xem phần so sánh bên dưới).

### Lưu trữ file (Storage)

Lưu trữ file (Storage): Supabase cung cấp dịch vụ Storage để lưu trữ các tệp (hình ảnh, video, tài liệu, v.v.) tương tự như Firebase Storage. Hệ thống Storage của Supabase sử dụng một đối tượng lưu trữ tương thích S3 làm backend. Một điểm đặc trưng: metadata file được lưu trong Postgres – mỗi tệp upload sẽ tạo một dòng trong bảng objects của Supabase, cho phép áp dụng cùng chính sách RLS bảo mật như dữ liệu khác trong ứng dụng Nhờ vậy, quyền truy cập file (xem/tải file) có thể kiểm soát chi tiết dựa trên user, tương tự như quyền truy cập dữ liệu bảng. Supabase Storage hỗ trợ upload dung lượng lớn với resume (tự động chia nhỏ và tiếp tục upload nếu gián đoạn) Ngoài ra còn có sẵn CDN tích hợp và xử lý hình ảnh (như thay đổi kích thước, crop) ngay trong nền tảng – tính năng mà Firebase cần dùng thêm extension bên ngoài mới có.

### Functions (Serverless)

Functions (Serverless): Tương tự Firebase Cloud Functions, Supabase cung cấp Edge Functions để viết hàm serverless chạy trên hạ tầng Supabase. Edge Functions của Supabase chạy trên nền Deno runtime (JavaScript/TypeScript) và được triển khai phân tán tại các edge location để giảm độ trễ người dùng Lập trình viên có thể viết các hàm TS và deploy qua CLI; các hàm này có thể được gọi qua HTTP hoặc thiết lập trigger theo lịch. Điểm mạnh là Edge Functions có quyền truy cập trực tiếp vào database Postgres của Supabase (thông qua SDK hoặc query trực tiếp) Điều này giúp viết các logic tùy chỉnh liên quan dữ liệu (ví dụ xác thực webhook, xử lý transaction phức tạp, gửi email…) một cách dễ dàng, tích hợp chặt với DB mà không cần trung gian. (Firebase Functions thì tích hợp sâu với các sự kiện Firebase/GCP hơn – xem so sánh). Hiện tại Edge Functions của Supabase hỗ trợ chính TypeScript (Deno), chưa đa dạng ngôn ngữ như Firebase Cloud Functions

### API & công cụ phát triển

**API & công cụ phát triển**: Mọi project Supabase khi khởi tạo sẽ tự động có một RESTful API cho database của bạn thông qua PostgREST. Toàn bộ các bảng và chức năng Postgres (function) có thể truy cập qua API REST này (với cơ chế query bằng querystring khá linh hoạt). Ngoài ra Supabase cũng cung cấp API GraphQL (dựa trên extension pg_graphql) cho phép query dữ liệu qua GraphQL nếu cần. Nhờ auto-API này, bạn có thể xây dựng ứng dụng mà không cần viết server thủ công cho các thao tác CRUD cơ bản – tương tự triết lý "không cần backend" của Firebase. Supabase cũng có bảng điều khiển (Supabase Studio) trực quan trên nền web – bạn có thể duyệt và quản lý các bảng dữ liệu, chạy câu SQL, thiết lập RLS, quản lý người dùng xác thực… ngay trên giao diện đồ họa thân thiện. Tất cả các client library phổ biến đều được Supabase hỗ trợ: hiện có SDK chính thức cho JavaScript/TypeScript, Flutter (Dart), Swift (iOS) và cộng đồng đóng góp SDK cho nhiều ngôn ngữ khác như Python, C#, Kotlin, Rust…. Ngoài ra do Supabase hỗ trợ REST và GraphQL, về lý thuyết bất kỳ ngôn ngữ nào biết gọi HTTP request đều có thể dùng Supabase API dễ dàng.

### Điểm mạnh của Supabase

- **Mã nguồn mở & tránh khóa vendor**: Toàn bộ nền tảng Supabase (Postgres, Auth (GoTrue), Realtime, Storage, Dashboard…) đều là mã nguồn mở dưới các giấy phép permissive. Điều này có nghĩa là bạn có thể tự host Supabase trên server của riêng mình (qua Docker) hoặc triển khai trên cloud tùy chọn, không bắt buộc dùng dịch vụ của Supabase Inc. Kiến trúc Supabase cũng đảm bảo tính di động dữ liệu: vì sử dụng Postgres tiêu chuẩn nên bạn có thể sao lưu, xuất dữ liệu và thậm chí di chuyển sang hệ quản trị khác khi cần khá dễ dàng. Nếu dùng Supabase Cloud, bạn vẫn có thể chuyển đổi sang tự host hoặc ngược lại mà không cần viết lại code ứng dụng (miễn là tuân thủ các API tương ứng). Đây là lợi thế lớn so với Firebase (mã nguồn đóng và phụ thuộc hạ tầng Google).

- **Sức mạnh của SQL và hệ sinh thái Postgres**: Nhờ dùng PostgreSQL, Supabase hỗ trợ đầy đủ SQL cho phép truy vấn và tính toán dữ liệu phức tạp (ví dụ join nhiều bảng, aggregate phức tạp) – điều mà Firestore gặp khó khăn. Bạn có thể thiết kế quan hệ ràng buộc (foreign key) và tận dụng ràng buộc toàn vẹn để đảm bảo tính đúng đắn của dữ liệu. Ngoài ra, Postgres có hệ sinh thái phong phú với nhiều extension hữu ích: Supabase đã hỗ trợ sẵn pg_trgm (tìm kiếm toàn văn), PostGIS (dữ liệu địa lý), và mới đây là pgvector để lưu vector embeddings phục vụ các ứng dụng AI/ML. Điều này có nghĩa Supabase có thể dễ dàng hỗ trợ các tính năng nâng cao như tìm kiếm ngữ nghĩa, gợi ý dựa trên ML – vốn không thể làm trực tiếp trên Firebase. Supabase trao cho developer toàn quyền truy cập database (qua SQL, console, psql) nên rất linh hoạt trong việc quản trị và tối ưu, tương tự như khi dùng một PostgreSQL thuần túy.

- **Dễ sử dụng với developer hiện đại**: Mặc dù dựa trên SQL, Supabase hướng đến việc đơn giản hóa tối đa trải nghiệm như Firebase. Chỉ cần vài phút là bạn có thể tạo xong một cơ sở dữ liệu Postgres online và sử dụng ngay API mà không phải cấu hình máy chủ. Giao diện Supabase Studio giúp thực hiện nhiều việc (như tạo bảng, thêm cột, thiết lập auth) một cách trực quan mà không cần chạy lệnh thủ công. Đối với những developer quen làm việc với bảng tính hoặc SQL, mô hình bảng và hàng của Supabase khá trực quan, dữ liệu có cấu trúc rõ ràng (schema) nên dễ hình dung hơn so với Firestore (phải tổ chức collection/document lồng nhau). Supabase cũng tích hợp tốt với các công cụ phát triển web hiện đại (đã có hướng dẫn tích hợp Next.js, Remix, Flutter, …). Bên cạnh đó, Supabase có cộng đồng năng động hỗ trợ (trên Discord, GitHub) do tính mở của nền tảng, giúp giải đáp và chia sẻ kinh nghiệm nhanh chóng.

- **Chi phí minh bạch, có thể tối ưu**: Supabase áp dụng mô hình giá theo tầng cố định (tiered pricing) khá dễ dự đoán. Có gói Free hào phóng cho dự án nhỏ, và gói Pro (~25$/tháng) bao gồm lượng tài nguyên cố định, nếu vượt sẽ tính thêm theo mức sử dụng tài nguyên (dung lượng lưu trữ, băng thông). Quan trọng là Supabase không tính tiền theo số lần request – bạn gọi API bao nhiêu cũng không bị tính phí riêng (vì về bản chất chỉ tính trên tài nguyên DB phía sau). Điều này giúp tránh được chi phí tăng đột biến khi ứng dụng scale, khác với Firebase (sẽ đề cập bên dưới). Ngoài ra, do có thể tự host, các công ty có thể lựa chọn tự vận hành Supabase trên cloud riêng để tiết kiệm chi phí khi đạt quy mô lớn, hoặc đáp ứng yêu cầu tuân thủ (compliance) mà bản Firebase không đáp ứng.

### Hạn chế của Supabase

- **Chưa có dịch vụ Hosting front-end**: Hiện tại Supabase không cung cấp hosting cho ứng dụng web hoặc mobile. Nghĩa là nếu bạn xây dựng frontend (React, Angular, mobile app), bạn sẽ cần dùng dịch vụ khác (như Vercel, Netlify hoặc máy chủ riêng) để host phần giao diện. Đội ngũ Supabase xác nhận chưa hỗ trợ static hosting (có thể bổ sung trong tương lai). Điều này khác với Firebase (có Firebase Hosting giúp triển khai web dễ dàng). Tuy không ảnh hưởng đến chức năng backend, đây vẫn là một mảnh ghép thiếu nếu bạn muốn “tất-cả-trong-một” như Firebase.

- **Hệ sinh thái tính năng hẹp hơn Firebase**: Supabase tập trung chủ yếu vào các tính năng backend cốt lõi (DB, auth, storage, function). Nền tảng này không có sẵn các dịch vụ khác như: dịch vụ push notification (Firebase Cloud Messaging), phân tích sự cố (Crashlytics), analytics tích hợp, hay machine learning framework… Do đó, nếu ứng dụng của bạn cần những dịch vụ này, bạn phải tích hợp thêm các dịch vụ bên ngoài (ví dụ OneSignal cho push, Sentry cho crash report, v.v.). Supabase hướng tới việc làm tốt phần backend cơ sở, còn các tính năng ngoài phạm vi database thì developer tự chọn giải pháp. Cách tiếp cận này khiến Supabase giống một "dịch vụ database + auth" hơn là một nền tảng toàn diện như Firebase. Thực tế, có ý kiến cho rằng Supabase hiện tại chưa thể thay thế hoàn toàn Firebase để xây dựng toàn bộ ứng dụng, mà chủ yếu như một giải pháp thay thế cho phần database (ví dụ thay MongoDB + Auth0).

- **Đòi hỏi kiến thức SQL và thiết kế CSDL**: Một điểm cần lưu ý, Supabase là database quan hệ, do đó bạn sẽ cần thiết kế bảng, cột, khóa chính/phụ và viết câu truy vấn SQL. Đối với những developer chưa quen SQL hoặc muốn nhanh chóng dựng prototype không quan tâm mô hình dữ liệu, điều này có thể tạo chút rào cản. Firebase với NoSQL schema-less cho phép linh hoạt thêm trường dữ liệu mới mà không cần migrate schema, trong khi Supabase buộc bạn quy hoạch trước cấu trúc (dù có thể thay đổi sau nhưng phức tạp hơn). Tương tự, việc cấu hình bảo mật trong Supabase (RLS) yêu cầu hiểu SQL để viết policy đúng, thay vì dùng rule dạng JSON như Firebase. Nhìn chung, Supabase phù hợp với đội ngũ đã có kinh nghiệm về SQL/relational; còn nếu bạn thuần frontend và chưa rành backend, thời gian đầu học cách dùng Supabase có thể lâu hơn một chút so với Firebase.

- **Khả năng mở rộng (scalability) còn hạn chế ở mức rất lớn**: Supabase dùng một instance Postgres cho mỗi project, do đó khả năng mở rộng phụ thuộc nhiều vào việc nâng cấp cấu hình máy chủ (CPU, RAM) hoặc thiết lập read-replica để chia tải đọc. Với ứng dụng thông thường, Postgres có thể phục vụ hàng ngàn người dùng đồng thời mà không vấn đề. Tuy nhiên, trong trường hợp ứng dụng cần quy mô cực lớn hoặc đột biến (hàng chục triệu người dùng), việc scale-out (chia nhỏ database) không tự động như Firestore. Đội ngũ Supabase có hỗ trợ tùy chọn read replicas và tối ưu hạ tầng, nhưng lập trình viên có thể cần tối ưu thủ công (viết index, tối ưu query) để đảm bảo hiệu suất khi dữ liệu rất lớn. Mặt khác, Firebase Firestore được Google tự động scale hạ tầng phía sau (phân tán dữ liệu), nên với khối lượng người dùng rất lớn bạn ít phải lo về hạ tầng hơn – đổi lại bạn phải chấp nhận hạn chế của NoSQL. Tóm lại, Supabase đủ khả năng mở rộng cho đa số ứng dụng startup cỡ vừa và nhỏ, nhưng với ứng dụng siêu lớn hoặc yêu cầu multi-region active-active, Firebase/GCP có lợi thế hơn về mặt hạ tầng tự động.

- **Tính năng offline và realtime chưa "mượt" bằng Firebase**: Như đã đề cập, Supabase hiện chưa hỗ trợ cache offline tự động trên client. Nếu ứng dụng di động mất mạng, bạn sẽ phải tự lưu tạm dữ liệu (ví dụ trong SQLite local) rồi tự đồng bộ khi có mạng – trong khi Firebase SDK làm điều này tự động cho bạn. Ngoài ra, realtime của Supabase dựa trên thay đổi trong database nên linh hoạt ở mức bảng/hàng, nhưng không linh động subscribe theo truy vấn tùy ý như Firestore. (Ví dụ: Firebase cho phép lắng nghe kết quả query "tất cả document thỏa điều kiện X" và tự cập nhật khi có document mới thỏa điều kiện; Supabase thì thường subscribe ở cấp bảng rồi client tự lọc). Bên cạnh đó, Firebase Realtime Database có tính năng onDisconnect() giúp xử lý presence đơn giản, còn Supabase đã bổ sung presence nhưng vẫn đang hoàn thiện. Nhìn chung, nếu ứng dụng của bạn cần offline-first mạnh (điện thoại hay mất mạng, cần đồng bộ nền), Firebase có lợi thế về độ mature của giải pháp. Supabase realtime mạnh ở tính nhất quán dữ liệu (do gắn liền transaction DB) và phù hợp cho ứng dụng yêu cầu dữ liệu real-time chính xác tuyệt đối, nhưng về độ “mượt” (khôi phục kết nối, xử lý xung đột offline) thì Firebase nhỉnh hơn.

- **Hệ sinh thái còn mới, cộng đồng nhỏ hơn**: Dù phát triển nhanh, Supabase vẫn là người đến sau so với Firebase. Cộng đồng Supabase (~1.7 triệu dev đăng ký tính đến 2025) đang lớn mạnh, tuy nhiên quy mô vẫn nhỏ hơn nhiều so với hàng triệu lập trình viên đã quen dùng Firebase. Một số vấn đề hay gặp có thể chưa có nhiều bài viết/blog hướng dẫn bằng tiếng Việt, và bạn có thể phải tra GitHub Discussions hoặc Discord để tìm trợ giúp. Tài liệu Supabase tuy đầy đủ nhưng sẽ không có kho kiến thức dồi dào như Firebase (với StackOverflow, YouTube, Medium tràn ngập). Ngoài ra, do Supabase còn trẻ, đôi khi có tính năng thay đổi (dù core API cam kết ổn định ở mức GA). Tất nhiên nhược điểm này sẽ giảm dần khi Supabase trưởng thành hơn.

## Firebase

**Firebase** là một nền tảng Backend-as-a-Service do Google cung cấp, giúp các lập trình viên xây dựng ứng dụng mà không cần tự quản lý server. Firebase ban đầu ra mắt năm 2011 như một dịch vụ cơ sở dữ liệu realtime NoSQL cho mobile, và được Google mua lại năm 2014. Kể từ đó, Firebase đã phát triển thành một nền tảng toàn diện bao gồm nhiều dịch vụ backend tích hợp chặt chẽ với hệ sinh thái Google Cloud. Hiện nay, Firebase cung cấp hầu hết mọi thứ cho phần backend của ứng dụng: cơ sở dữ liệu, xác thực, lưu trữ file, host website, functions serverless, phân tích, thông báo đẩy, test… – tất cả dưới dạng dịch vụ được Google quản lý hoàn toàn (fully-managed). Mục tiêu của Firebase là giúp developer "focus vào trải nghiệm người dùng, còn việc server đã có Google lo".

## Tính năng chính

### Realtime Database

Công nghệ nền tảng: Trọng tâm của Firebase là các dịch vụ database NoSQL realtime. Firebase hiện có hai lựa chọn database chính

- **Cloud Firestore**: Đây là database NoSQL mới (thay thế phần lớn cho Realtime DB cũ). Firestore lưu trữ dữ liệu dưới dạng document (tương tự JSON) trong collection. Mô hình này schema-less (không yêu cầu cấu trúc cố định, các document trong cùng collection có thể khác trường). Firestore tự động tạo chỉ mục cho các field để hỗ trợ truy vấn trên thuộc tính, và cho phép truy vấn theo điều kiện, sắp xếp, phân trang khá linh hoạt. Tuy nhiên, Firestore không hỗ trợ join giữa các collection một cách tự nhiên (vì là NoSQL) – nếu cần kết hợp dữ liệu, thường phải tổ chức dữ liệu dạng lồng hoặc thực hiện nhiều query rồi tổng hợp ở client. Firestore được thiết kế cho đồng bộ realtime: client có thể đăng ký lắng nghe một collection hoặc một query, và sẽ nhận cập nhật ngay khi có thay đổi. Đặc biệt, hỗ trợ offline là tính năng mạnh: SDK Firebase sẽ cache dữ liệu trên device (những dữ liệu đã fetch hoặc đánh dấu sync), cho phép app đọc/ghi offline và tự đồng bộ khi có kết nối lại. Firestore cũng đảm bảo tính nhất quán cuối cùng (eventual consistency) trong môi trường phân tán nhiều vùng.

- **Realtime Database (RTDB)**: Đây là database NoSQL đời đầu của Firebase, lưu trữ dữ liệu dạng cây JSON. RTDB cho phép đồng bộ realtime tương tự Firestore nhưng với cấu trúc đơn giản hơn (toàn bộ dữ liệu là một JSON tree lớn). RTDB thích hợp cho những ứng dụng dữ liệu không quá phức tạp, yêu cầu realtime tốc độ cao (ví dụ ứng dụng chat, status online đơn giản). So với Firestore, RTDB có số lượng tính năng hạn chế hơn: không có query linh hoạt (chỉ query được theo key hoặc giá trị đơn giản), không có cấu trúc collection/document mà chỉ là node JSON. Hiện Google định hướng Firestore là lựa chọn chính, nhưng RTDB vẫn tồn tại để hỗ trợ các trường hợp cần độ trễ rất thấp hoặc cấu trúc dữ liệu linh hoạt không cần schema. RTDB cũng hỗ trợ cache offline tương tự Firestore.

### Xác thực người dùng (Auth)

- **Authentication**: Firebase Authentication là dịch vụ xác thực người dùng đầy đủ tính năng. Nó hỗ trợ tất cả các phương thức đăng nhập phổ biến: Email/Mật khẩu, OAuth qua bên thứ ba (Google, Facebook, Twitter, GitHub, v.v.), Số điện thoại (OTP SMS), và thậm chí Anonymous (tài khoản ẩn danh không cần thông tin đăng nhập, thường để người dùng dùng thử tạm thời). Firebase Auth cũng cho phép bật xác thực đa yếu tố (MFA) dễ dàng. SDK Firebase cung cấp sẵn UI và logic để thực hiện đăng nhập, nên chỉ cần vài dòng code là ứng dụng có màn hình đăng nhập hoạt động. Điểm mạnh là Firebase Auth tích hợp chặt chẽ với các dịch vụ Firebase khác: ví dụ, Firestore và Storage có thể dựa vào trạng thái đăng nhập (UID người dùng) để quyết định quyền truy cập tài nguyên thông qua Security Rules (xem bên dưới). Bên cạnh đó, Firebase Auth có sẵn tích hợp với Google Identity Platform cho các tính năng doanh nghiệp như SAML, OIDC (có thể yêu cầu nâng cấp gói). Nói chung, Firebase Auth dễ sử dụng cho dev frontend và đủ mạnh cho các yêu cầu xác thực thông thường.

### Security Rules (Bảo mật dữ liệu)

- **Security Rules (Bảo mật dữ liệu)**: Đối với Firestore và Storage, Firebase sử dụng một cơ chế bảo mật gọi là Firebase Security Rules. Đây là các quy tắc dưới dạng JSON-like script (cú pháp tương tự JavaScript) cho phép định nghĩa ai được đọc/ghi vào phần nào của dữ liệu, dựa trên thông tin người dùng và nội dung yêu cầu. Mỗi dịch vụ (Firestore, Storage, RealtimeDB) có bộ rule riêng, và chúng được thực thi trên server Google mỗi khi có yêu cầu từ client. Ví dụ, bạn có thể viết rule cho Firestore kiểu: _"cho phép ghi vào tài liệu nếu request.auth.uid == userId trong tài liệu đó"_. Rules rất mạnh trong việc kiểm soát truy cập động (dựa vào data và user), giúp bạn bảo vệ dữ liệu ngay ở phía backend mà không cần viết server xác minh. Tuy nhiên, viết rule phức tạp có thể hơi khó và dễ sai nếu không cẩn thận. Về độ chi tiết, Security Rules thực thi ở tầng service (document hoặc file), không chi tiết đến cột/hàng như RLS của Supabase ￼. Dù vậy, nó dễ tiếp cận hơn với developer frontend vì dùng ngôn ngữ gần gũi và có công cụ giả lập để thử nghiệm.

### Hosting (Web Hosting)

- **Hosting**: Firebase cung cấp Firebase Hosting – dịch vụ hosting web tĩnh miễn phí và tiện lợi. Với Firebase Hosting, bạn có thể triển khai nhanh chóng các website tĩnh, ứng dụng single-page (SPA) hoặc thậm chí kết hợp nội dung động qua Cloud Functions. Firebase Hosting tự động cấu hình HTTP/2 + SSL (có chứng chỉ HTTPS miễn phí) và dùng CDN của Google trên toàn cầu để phân phối nội dung nhanh chóng. Chỉ với lệnh firebase deploy, bạn có thể đưa ứng dụng web (HTML, CSS, JS, media…) lên Internet trong vài giây. Hosting cũng hỗ trợ rewrite request đến Cloud Functions hoặc các dịch vụ khác, cho phép chạy backend API hoặc server-side rendering cho ứng dụng web. Tính năng hosting tích hợp này là lợi thế lớn của Firebase, giúp bạn có giải pháp end-to-end (frontend + backend) trong một nền tảng, trong khi Supabase không có tính năng tương tự.

### Cloud Functions (Serverless)

- **Cloud Functions**: Firebase Cloud Functions là môi trường thực thi serverless do Google Cloud cung cấp, tích hợp sẵn trong Firebase để viết các logic backend tùy chỉnh. Bạn có thể viết hàm bằng JavaScript/TypeScript (chạy trên Node.js runtime). Gần đây, thế hệ Cloud Functions 2nd gen còn hỗ trợ Python, Go, Java… (thực chất là dùng Google Cloud Functions chung). Những hàm này có thể được kích hoạt bởi nhiều loại sự kiện: HTTP request, lịch hẹn giờ, sự kiện ghi/xóa Firestore, sự kiện xác thực (user đăng ký mới), sự kiện upload file Storage, sự kiện Pub/Sub,…. Khả năng tích hợp sự kiện phong phú là ưu điểm nổi trội của Firebase Functions – ví dụ bạn có thể viết một hàm tự chạy khi người dùng đăng ký mới để gởi email welcome, hoặc hàm mỗi khi có file upload để chuyển đổi hình ảnh. Cloud Functions chạy trên hạ tầng Google nên tự động scale-out khi tải tăng, nhưng cũng có thể gặp độ trễ cold start (khởi động chậm) nếu hàm ít được gọi. So với Supabase Edge Functions (chỉ hỗ trợ TS/Deno), Firebase Functions đa dạng ngôn ngữ hơn và tích hợp mạnh với toàn bộ hệ sinh thái Google Cloud (bạn có thể truy cập bất kỳ dịch vụ GCP nào từ code của mình). Tuy nhiên, nếu chỉ so phần backend gần database thì Functions của Firebase không có kết nối trực tiếp database như Supabase (bạn phải gọi qua SDK hoặc REST như một client bình thường).

### Storage (Lưu trữ tệp)

- **Storage**: Firebase cung cấp Cloud Storage for Firebase, thực chất là một lớp tiện ích trên Google Cloud Storage (GCS). Nó cho phép ứng dụng mobile/web lưu trữ các tệp lớn (hình ảnh, video, audio…). Điểm mạnh là Firebase Storage tích hợp với Firebase Auth và Security Rules: bạn có thể dễ dàng thiết lập rule như “chỉ user nào auth mới được upload” hoặc “user chỉ được download file thuộc về họ”… tương tự cách bảo mật Firestore. Dữ liệu file được lưu trong bucket GCS, rất bền bỉ (đã được Google sao lưu). Firebase Storage SDK hỗ trợ upload/download với resumable upload (tiếp tục khi mạng nối lại) và theo dõi tiến trình. Về tính năng xử lý file: Firebase Storage không có sẵn chức năng biến đổi hình ảnh hay CDN nội bộ – nhưng Google cung cấp các Firebase Extensions để thêm (ví dụ extension Image Resizer để tự động tạo thumbnail khi upload). Nói chung, Firebase Storage đơn giản, đáng tin cậy, phù hợp cho ứng dụng cần lưu user-generated content và chia sẻ dễ dàng thông qua URL có quyền hạn. (Supabase Storage cũng tương tự về chức năng, khác ở chỗ dùng RLS để bảo mật).

### Các dịch vụ khác trong hệ sinh thái

- **Các dịch vụ khác trong hệ sinh thái**: Firebase thực sự mạnh nhờ hệ sinh thái toàn diện vượt xa database. Một số dịch vụ đáng chú ý khác gồm: Firebase Cloud Messaging (FCM) – dịch vụ gửi thông báo đẩy (push notifications) miễn phí đến thiết bị Android/iOS hoặc web browser; Remote Config – cho phép lưu cấu hình từ xa và phân phối real-time đến app (thường dùng để feature toggle, A/B testing); Firebase Crashlytics – công cụ thu thập và báo cáo sự cố/crash của app theo thời gian thực; Firebase Performance Monitoring – theo dõi hiệu năng (thời gian load, độ trễ) của app; Google Analytics cho Firebase – tích hợp sẵn Google Analytics để thu thập sự kiện và hành vi người dùng trong app; Firebase Test Lab – hạ tầng test tự động trên nhiều thiết bị thật cho ứng dụng Android/iOS. Tất cả các dịch vụ này đều miễn phí ở mức độ nào đó và dễ dàng kích hoạt chỉ với vài bước. Chính vì có nhiều mảnh ghép như vậy, Firebase trở thành lựa chọn hấp dẫn để xây dựng ứng dụng di động một cách nhanh chóng, tập trung vào trải nghiệm mà không tốn công tích hợp nhiều dịch vụ bên ngoài.

## Điểm mạnh của Firebase

- **Realtime và Offline xuất sắc**: Firebase được thiết kế cho ứng dụng realtime ngay từ đầu, do đó các tính năng đồng bộ thời gian thực của nó rất hoàn thiện và đáng tin cậy. Mọi thay đổi dữ liệu có thể được tự động đẩy đến tất cả client trong vòng mili-giây. Cùng với đó, khả năng hoạt động offline (nhất là trên thiết bị di động) của Firebase vượt trội: SDK tự động đồng bộ và giải quyết xung đột, giúp trải nghiệm người dùng mượt mà ngay cả khi mạng chập chờn. Đây là lợi thế mà Supabase (dựa trên Postgres) khó bắt kịp, đặc biệt trong các ứng dụng cần offline-first.

- **Dễ sử dụng, không đòi hỏi nhiều kiến thức backend**: Firebase hướng tới đối tượng front-end developers hoặc những nhóm nhỏ không có chuyên viên backend. Bạn không cần biết SQL hay quản lý server, chỉ cần dùng SDK Firebase là có thể đọc/ghi dữ liệu, xác thực người dùng một cách trực quan. Việc không cần định nghĩa schema trước (như Firestore) giúp bạn linh hoạt chỉnh sửa mô hình dữ liệu trong giai đoạn prototyping mà không phải migrate phức tạp. Tài liệu Firebase rất đầy đủ, ví dụ phong phú, và có nhiều hướng dẫn chính thức cũng như cộng đồng (StackOverflow, Reddit) hỗ trợ. Theo đánh giá, để bắt đầu với Firebase, bạn chỉ cần kiến thức cơ bản và vài giờ đọc docs là có thể làm được login, lưu trữ dữ liệu mà không gặp trở ngại lớn. Nhiều tác vụ phức tạp đã được Firebase đơn giản hóa thành các API gọn nhẹ (ví dụ: xác thực email/password chỉ vài dòng code, triển khai hosting một lệnh duy nhất). So với Supabase, Firebase thân thiện với người mới hơn, đánh đổi bằng việc ẩn đi một số chi tiết (như không cho truy vấn quá sâu như SQL).

- **Hệ sinh thái tính năng phong phú & tích hợp sẵn**: Như liệt kê ở trên, Firebase không chỉ có database mà còn có hàng loạt dịch vụ giá trị gia tăng (FCM, Crashlytics, Analytics, A/B testing…). Tất cả đều do Google cung cấp miễn phí hoặc miễn phí hạn mức và tích hợp tương thích với nhau. Ví dụ: bạn có thể dễ dàng gửi push notification đến người dùng ứng dụng của mình thông qua FCM mà không tốn phí; hoặc dùng Remote Config để thử nghiệm A/B một tính năng UI mới. Sự phong phú này giúp các nhóm nhỏ tiết kiệm rất nhiều thời gian thay vì phải tích hợp các dịch vụ bên thứ ba khác nhau. Về khía cạnh này thì Supabase chưa thể sánh kịp (phải kết hợp nhiều dịch vụ ngoài nếu cần chức năng tương tự).

- **Khả năng mở rộng cao, hạ tầng Google đáng tin cậy**: Firebase chạy trên hạ tầng cloud của Google (vốn nổi tiếng về hiệu năng và độ ổn định). Firestore được xây dựng để tự động mở rộng (auto-scale) – khi ứng dụng của bạn có nhiều người dùng hơn, Google sẽ tự động phân bổ thêm tài nguyên phía sau, bạn không cần lo việc nâng cấp server. Dịch vụ Firebase cũng đa vùng (multi-region): bạn có thể chọn đặt dữ liệu tại nhiều khu vực địa lý để tăng tốc độ truy cập và dự phòng (Firestore cung cấp tùy chọn multi-region mode). Nhờ đó, Firebase phù hợp cho các ứng dụng có tham vọng phục vụ hàng triệu người dùng trên toàn cầu – điều mà nếu tự quản lý database (như Postgres của Supabase) sẽ rất phức tạp. Hơn nữa, Firebase do Google quản lý nên bạn hưởng lợi từ các SLA cao, backup, bảo mật hạ tầng, và cập nhật dịch vụ liên tục. Tính tin cậy (reliability) và hiệu suất của Firebase trong môi trường production thực tế đã được kiểm chứng qua rất nhiều ứng dụng lớn.

- **Miễn phí khởi điểm, chi phí linh hoạt theo nhu cầu**: Firebase có gói Spark (miễn phí) với hạn mức đủ rộng rãi cho dự án nhỏ hoặc giai đoạn thử nghiệm: ví dụ Firestore cho 50k lượt đọc và 20k lượt ghi mỗi ngày miễn phí, 5GB lưu trữ file miễn phí, 10GB bandwidth, v.v. Đặc biệt Firebase cho phép tạo không giới hạn project ở gói miễn phí, thuận tiện cho thử nghiệm nhiều ý tưởng hoặc tách môi trường dev/test. Khi ứng dụng phát triển hơn, bạn chuyển sang gói Blaze (trả phí theo mức dùng) – chi phí sẽ tỉ lệ với tài nguyên sử dụng (số lần đọc/ghi, dung lượng, số người dùng, v.v.). Ưu điểm của mô hình này là rất tối ưu cho khởi đầu nhỏ: bạn hầu như không phải trả gì cho đến khi ứng dụng có người dùng thực sự, và khi đó chi phí tăng dần theo doanh thu/user. So với Supabase (phải trả tối thiểu $25/tháng cho bản Pro dù dùng ít), Firebase Blaze có thể rẻ hơn ở quy mô nhỏ. Ngoài ra, việc tích hợp thanh toán qua Google Cloud Platform giúp các công ty có thể dùng chung ngân sách và nhận hóa đơn từ Google dễ dàng.

- **Cộng đồng rộng lớn, tài liệu và hỗ trợ tốt**: Firebase có mặt trên thị trường hơn một thập kỷ nên đã hình thành một cộng đồng khổng lồ. Theo một số thống kê, có khoảng 1.5 triệu ứng dụng trên toàn thế giới đang sử dụng Firebase. Điều này đồng nghĩa với vô số câu hỏi đã được thảo luận trên StackOverflow, rất nhiều blog, video hướng dẫn, khóa học về Firebase. Google cũng đầu tư nhiều vào tài liệu chính thức, cung cấp hướng dẫn cho mọi SDK (Android, iOS, Web, Unity…). Nếu gặp vấn đề, bạn có thể dễ dàng tìm thấy giải pháp do người khác chia sẻ. Firebase cũng có kênh Google support (đặc biệt cho gói trả phí) và các specialist cộng đồng (Firebase Google Developer Experts) sẵn sàng hỗ trợ. Nhìn chung, học và sử dụng Firebase rất thuận lợi nhờ tài nguyên học tập và cộng đồng hỗ trợ dồi dào. Đây là lợi thế mà Supabase (cộng đồng ~1.7 triệu dev vào 2025) đang nỗ lực bắt kịp, nhưng vẫn cần thời gian.

## Hạn chế của Firebase

- **Mô hình NoSQL gây khó khăn cho dữ liệu phức tạp**: Điểm yếu cố hữu của Firebase (Firestore/RTDB) là thiếu cấu trúc quan hệ. Dữ liệu JSON/document linh hoạt, nhưng khi dữ liệu có mối quan hệ phức tạp (1-n, n-n) hoặc cần truy vấn liên bảng (join) thì rất khó thể hiện. Lập trình viên thường phải lưu trữ dư thừa (denormalize) dữ liệu để tránh join, dẫn đến dữ liệu có thể mâu thuẫn nếu không cập nhật hết các bản sao. Firestore cũng giới hạn truy vấn: không có join, không hỗ trợ truy vấn văn bản toàn văn, hạn chế thao tác trên nhiều collection một lúc, v.v. Điều này làm cho Firebase không phù hợp cho những ứng dụng cần báo cáo, thống kê phức tạp trên dữ liệu (ví dụ hệ thống BI, analytics trực tiếp trên DB). Nếu cố dùng Firebase cho bài toán phân tích phức tạp, bạn thường phải xuất dữ liệu sang BigQuery để xử lý – khá bất tiện. Tóm lại, tính linh hoạt schema là con dao hai lưỡi: dễ bắt đầu nhưng có thể trả giá khi ứng dụng lớn và cần dữ liệu có cấu trúc chặt chẽ.

- **Không đảm bảo toàn vẹn và ràng buộc dữ liệu**: Do Firestore/RTDB không có khái niệm schema, bạn không thể đặt ràng buộc (constraints) hay kiểm soát chặt chẽ kiểu dữ liệu ở tầng database. Mọi logic kiểm tra (ví dụ uniqueness, không null, tính toán phụ thuộc) phải làm ở phía ứng dụng hoặc Cloud Function. Nếu không cẩn thận, dữ liệu rất dễ "bừa bộn" hoặc lỗi mà database không ngăn chặn được. Với các hệ thống cần tính toàn vẹn cao (ví dụ tài chính, ngân hàng) thì mô hình Firebase không đáp ứng tốt bằng SQL. Ngoài ra, do không có transaction rộng (Firestore chỉ hỗ trợ transaction trong một collection giới hạn), các cập nhật liên quan nhiều loại dữ liệu có thể gặp vấn đề nếu xảy ra lỗi giữa chừng.

- **Phụ thuộc vào nền tảng Google (vendor lock-in)**: Sử dụng Firebase đồng nghĩa với việc bị khóa vào hệ sinh thái Google Cloud. Mã nguồn Firebase backend đóng nên bạn không thể tự host hay di chuyển dịch vụ Firebase sang nơi khác. Nếu sau này Google thay đổi chính sách giá hoặc giới hạn dịch vụ, bạn ít nhiều sẽ bị ảnh hưởng. Việc di chuyển dữ liệu ra khỏi Firebase cũng không dễ dàng – ví dụ Firestore không có công cụ xuất tất cả dữ liệu thành file JSON hay SQL dump thuận tiện (có thể dùng Data Export qua GCP nhưng phức tạp). Như một bài phân tích chỉ ra, Firebase hạn chế việc di trú dữ liệu khiến người dùng phụ thuộc vào nền tảng nhiều hơn. Đối với các công ty lo ngại vendor lock-in hoặc có yêu cầu triển khai on-premise, Firebase không phải lựa chọn phù hợp (Supabase lại có lợi thế mã nguồn mở ở điểm này).

- **Chi phí có thể tăng cao và khó dự đoán ở quy mô lớn**: Mặc dù khởi đầu rẻ, mô hình tính phí pay-as-you-go của Firebase có thể gây bất ngờ khi ứng dụng tăng trưởng. Firebase Firestore tính tiền theo số lượt đọc/ghi (ví dụ $0.06 cho 100k read), nghe có vẻ nhỏ nhưng khi app có hàng triệu người dùng, số đọc ghi có thể lên hàng tỷ, dẫn đến hóa đơn rất lớn. Đặc biệt, các dev mới có thể vô tình viết query kém tối ưu (đọc nhiều document không cần thiết) làm phí tăng đột biến. Việc dự đoán trước chi phí cũng khó, vì phụ thuộc vào hành vi người dùng. Trái lại, mô hình giá cố định theo tài nguyên (như Supabase) thì dễ dự trù ngân sách hơn. Ngoài ra, một số dịch vụ Firebase có phí riêng (ví dụ: gọi Cloud Functions tính tiền theo số lần và thời gian chạy, dùng Test Lab trả phí theo phút). Tóm lại, Firebase rất tiết kiệm cho MVP và ứng dụng nhỏ, nhưng nếu thành công lớn bạn phải chuẩn bị ngân sách đáng kể và tối ưu usage cẩn thận để tránh "hóa đơn sốc". (Tuy nhiên, cũng cần nói rằng chi phí tăng thường tương ứng với lượng người dùng tăng – nghĩa là bạn cũng đang thu được lợi ích tương ứng).

- **Giới hạn về giới hạn (quota) và cấu hình tùy chỉnh**: Do là dịch vụ managed, Firebase đặt ra nhiều giới hạn cố định: ví dụ Firestore giới hạn 1 write/giây trên mỗi document để tránh xung đột, giới hạn 10k write/giây mỗi database, mỗi truy vấn Firestore tối đa trả 1MB dữ liệu hoặc 10k document, v.v. Hầu hết các giới hạn này rất cao và khó chạm tới với ứng dụng thông thường, nhưng vẫn cần lưu ý trong một số trường hợp đặc thù. Ngoài ra, bạn không kiểm soát được cấu hình hạ tầng (như cấu hình CPU/RAM, server location chi tiết) – mọi thứ do Google quyết định. Điều này giúp đơn giản hóa, nhưng nếu bạn muốn tùy biến (ví dụ chọn database chạy trên CPU mạnh cho tính toán phức tạp) thì không thể. Với Supabase (hoặc tự host), bạn có thể tùy chọn cấu hình máy theo ý muốn.

- **Viết Security Rules phức tạp dễ lỗi**: Security Rules của Firebase ban đầu dễ viết cho các trường hợp đơn giản (ví dụ so khớp UID). Nhưng khi ứng dụng lớn, rules có thể trở nên rất phức tạp và dài dòng, dễ dẫn đến sai logic bảo mật nếu không test kỹ. Nhiều lập trình viên cho rằng debug rule khó hơn debug code thông thường do rule là ngôn ngữ khai báo, công cụ kiểm tra còn hạn chế. Ngược lại, Supabase dùng RLS ở tầng SQL – nếu đội ngũ quen SQL sẽ kiểm soát logic tốt hơn. Đây không hẳn là “lỗi” của Firebase, mà là cái giá phải trả cho việc không viết tầng backend server: bạn phải nhét logic vào rules. Với những dự án yêu cầu bảo mật cao, đôi khi người ta vẫn phải viết Cloud Functions trung gian thay vì phụ thuộc hoàn toàn vào Security Rules.

Tóm lại, Firebase lý tưởng cho các ứng dụng real-time, cần phát triển nhanh, và không quá phức tạp về dữ liệu quan hệ. Còn nếu ứng dụng đòi hỏi logic dữ liệu phức tạp, truy vấn nặng, hoặc muốn tự chủ hạ tầng, những hạn chế của Firebase có thể là rào cản – đó là lúc Supabase (hoặc các giải pháp backend khác) đáng để cân nhắc.

## So sánh Supabase và Firebase

Bảng dưới đây tóm tắt sự so sánh trực tiếp giữa Supabase và Firebase trên các khía cạnh quan trọng:

| Khía cạnh              | Supabase                                                                    | Firebase                                                                     |
| ---------------------- | --------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| **CSDL**               | **PostgreSQL (SQL, ACID)**, schema quan hệ, JOIN/transaction mạnh           | **NoSQL** (Cloud Firestore, Realtime DB), document/JSON, không JOIN          |
| **Realtime & Offline** | Realtime từ thay đổi Postgres (WS); **offline tự xử lý**                    | Realtime tích hợp sâu; **offline-first** tự động (cache/sync)                |
| **Auth**               | Email/Password, OAuth, OTP, Passwordless; **RLS** (Row-Level Security) ở DB | Email/Password, OAuth, Phone, **Anonymous**; **Security Rules** theo dịch vụ |
| **Storage**            | S3-compatible, metadata trong Postgres ⇒ áp **RLS**; có CDN & xử lý ảnh     | Cloud Storage (GCS), bảo vệ bằng **Storage Rules**; xử lý ảnh qua Extensions |
| **Hosting**            | **Không có** hosting frontend (dùng Vercel/Netlify/CF Pages…)               | **Firebase Hosting**: CDN + SSL, deploy 1 lệnh, rewrite tới Functions        |
| **Functions**          | **Edge Functions (Deno/TS)**, gần DB, triển khai edge                       | **Cloud Functions** (Node/TS + nhiều ngôn ngữ gen2), nhiều trigger GCP       |
| **SDK**                | JS/TS, Flutter, Swift + REST/GraphQL; cộng đồng bổ sung nhiều ngôn ngữ      | Android, iOS, Web, C++/Unity, Admin SDK (Node/Java/Python/Go…)               |
| **Mở & tự host**       | **Mã nguồn mở**, **tự host** dễ (Docker/K8s); tránh lock-in                 | Dịch vụ **đóng**, **không tự host**; gắn chặt hệ sinh thái Google            |
| **Giá**                | Free hạn mức; **Pro $25/tháng** (không tính theo số request)                | Free (Spark); **Blaze trả theo dùng** (read/write, invocations, băng thông)  |
| **Mở rộng**            | Scale theo instance Postgres + read-replica; cần tối ưu SQL                 | **Auto-scale** quản lý bởi Google; multi-region, SLA cao                     |
| **Phù hợp khi**        | Cần SQL/quan hệ, kiểm soát dữ liệu, tránh lock-in, self-host                | Cần phát triển nhanh, realtime + offline mượt, hệ sinh thái “all-in-one”     |

## Kết luận

Supabase và Firebase đều là những lựa chọn mạnh mẽ để tăng tốc phát triển ứng dụng mà không cần lo nhiều về backend truyền thống. Firebase tỏa sáng với sự tiện dụng, ổn định và hệ sinh thái “tất cả trong một” – rất phù hợp cho các ứng dụng realtime, di động, hoặc dự án cần triển khai nhanh, ít nhân lực backend. Ngược lại, Supabase nổi bật nhờ tính mở, sức mạnh của SQL và khả năng kiểm soát dữ liệu – thích hợp cho các dự án cần mô hình dữ liệu phức tạp, muốn tránh khóa vendor và tận dụng hệ sinh thái nguồn mở. Việc lựa chọn phụ thuộc vào ưu tiên của đội ngũ: nếu coi trọng mở nguồn, linh hoạt SQL và chấp nhận tự quản lý nhiều hơn, Supabase là lựa chọn hấp dẫn; nếu ưu tiên sự đơn giản, tích hợp sẵn và khả năng mở rộng tự động thì khó có nền tảng nào qua mặt được Firebase.
